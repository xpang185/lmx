# LMX Specification

> LLM Executables — Unix programs powered by LLMs

## Philosophy

LMX programs are standard Unix commands that happen to use an LLM internally. They follow the same contract as any Unix program:

- **Stateless**: input in, output out. No hidden internal state between invocations.
- **Composable**: read from stdin or args, write results to stdout, errors to stderr.
- **Exit codes**: 0 = success, non-zero = failure.
- **May mutate files**: like any program, but the command itself is a stateless transform.
- **Text is the interface**: stdout is data, stderr is diagnostics.

From the user's perspective, an LMX program is indistinguishable from `grep`, `jq`, or `curl`. The LLM is an implementation detail.

```bash
# These all look and work like normal commands
summarize "some text"
echo "long article" | translate --lang fr
curl https://example.com | extract-emails | sort -u
cat report.md | summarize --max-chars 200 --format bullets | translate --lang de
```

## Architecture

### Components

```
┌─────────────────────────────────────────────────┐
│  User invokes: summarize --max-chars 200 "text" │
│                                                 │
│  ┌──────────┐    ┌──────────┐    ┌───────────┐  │
│  │summarize │───>│ lmx run  │───>│ Pi SDK    │  │
│  │(shim)    │    │(runner)  │    │(LLM call) │  │
│  └──────────┘    └──────────┘    └───────────┘  │
│                       │                         │
│                  ┌────┴────────┐                │
│                  │ Pi Auth +   │                │
│                  │ ModelRegistry│               │
│                  └─────────────┘                │
└─────────────────────────────────────────────────┘
```

**`lmx`** — The runner/system. Handles model resolution, auth, param parsing and validation, context assembly, Pi invocation, and stdout/stderr/exit codes. The user never interacts with `lmx` directly.

**LMX program** — A folder containing a prompt, config, and optional scripts. Exposed to the user as a normal executable command.

**Pi SDK** — The LLM execution layer. Called via `createAgentSession()` with `systemPromptOverride` (the LMX.md content), `SessionManager.inMemory()` (stateless), and optional tools like `createBashTool()` for programs that need to run companion scripts.

**Pi Auth** — `AuthStorage.create()` reads `~/.pi/agent/auth.json` and environment variables. LMX reuses Pi's auth and model setup directly. No separate LMX auth layer.

### Pi-Only Runtime

LMX v1 is Pi-only. The runner uses Pi for auth, model resolution, and execution. This gives:

- **Zero bootstrap overhead**: No extra agent wrapper, reply tags, or tool preamble. The LMX.md IS the system prompt.
- **Auth for free**: `AuthStorage.create()` reads the same credentials Pi already manages.
- **Model resolution for free**: `ModelRegistry` resolves providers and handles failover.
- **Stateless execution**: `SessionManager.inMemory()` — no session files, no persistence.
- **Per-program tool selection**: Each LMX program can declare exactly which Pi tools it needs (e.g., `bash` for running companion scripts, or none at all).
- **Clear boundary**: Future OpenClaw integration can be added later without changing the core LMX runtime contract.

### Pi Invocation (Minimal)

```typescript
import { createAgentSession, SessionManager, DefaultResourceLoader, createBashTool } from "@mariozechner/pi-coding-agent";
import { getModel } from "@mariozechner/pi-ai";
import { AuthStorage, ModelRegistry } from "@mariozechner/pi-coding-agent";

const authStorage = AuthStorage.create();
const modelRegistry = ModelRegistry.create(authStorage);
const model = getModel(provider, modelId);

// Resolve {baseDir} in prompt content
const prompt = lmxContent.replace(/\{baseDir\}/g, programDir);

const loader = new DefaultResourceLoader({
  systemPromptOverride: () => prompt,
  skillsOverride: () => ({ skills: [], diagnostics: [] }),
  agentsFilesOverride: () => ({ agentsFiles: [] }),
});
await loader.reload();

const { session } = await createAgentSession({
  model,
  sessionManager: SessionManager.inMemory(),
  authStorage,
  modelRegistry,
  resourceLoader: loader,
  tools: needsBash ? [createBashTool(programDir)] : [],
  customTools: [],
});

let output = "";
session.subscribe((event) => {
  if (event.type === "message_update" && event.assistantMessageEvent.type === "text_delta") {
    output += event.assistantMessageEvent.delta;
  }
});

await session.prompt(assembledContext);
process.stdout.write(output);
```

## LMX Program Structure

```
~/.lmx/programs/summarize/
├── LMX.md              # Unstructured instructions / system prompt
├── config.yaml         # Structured config: metadata, params, model defaults, tools
├── help.txt            # Auto-generated by `lmx build`
├── summarize           # The executable shim (auto-generated by `lmx build`)
├── bench/              # Benchmarks
│   ├── bench.yaml      # Test cases and assertions
│   └── results/        # Per-model benchmark results
│       ├── anthropic_claude-sonnet-4-6.yaml
│       └── openai_gpt-5.4.yaml
├── scripts/            # Optional companion scripts
│   └── preprocess.py
└── ... other files
```

### LMX.md

The system prompt. This is what the LLM sees as its instructions. Uses `{baseDir}` to reference the program's directory (resolved by the runner before injection).

`LMX.md` is unstructured instructions only. Do not duplicate machine-readable metadata here. Name, description, params, tools, and other structured settings live in `config.yaml`.

```markdown
# Summarize

You are a summarization tool. Given input text, produce a concise summary.

If companion preprocessing is needed:
`python3 {baseDir}/scripts/preprocess.py`

Output only the summary. No preamble, no commentary, no markdown formatting unless requested.
```

The LMX.md author does NOT reference params in the instructions. The runner appends param context automatically.

### config.yaml

Single source of truth for structured config: metadata, parameters, model configuration, and runtime options.

```yaml
name: summarize
description: "Summarize text or documents concisely"
version: "1.0.0"

# Model configuration
default_model: openai-codex/gpt-5.4
min_bench_score: 0.80

# Tools this program needs
# LMX owns this schema; the runner maps it to Pi SDK tool constructors.
# Options: bash, read, none
tools:
  - bash

# Program-specific parameters
params:
  max-chars:
    description: "Maximum characters in the summary output"
    type: number
    default: 500
  format:
    description: "Output format"
    type: string
    enum: [paragraph, bullets, oneliner]
    default: paragraph
  instructions:
    description: "Additional instructions for the summarization"
    type: string
```

LMX does NOT inherit Pi's full config format. `config.yaml` is an LMX schema. The runner translates relevant fields into Pi SDK options. If LMX later needs more Pi-specific knobs, add them explicitly as small fields instead of mirroring Pi wholesale.

### The Executable Shim

Auto-generated by `lmx build`.

On Unix, every program gets the standard executable shim:

```bash
#!/bin/bash
lmx run "$(dirname "$0")" "$@"
```

The file is named after the program (no `.sh` extension). `chmod +x` by `lmx build`.

On Windows, `lmx build` also generates a sibling `<program>.cmd` wrapper so the program can be invoked from Command Prompt or PowerShell while keeping the same `lmx run <program-dir>` behavior.

### help.txt

Auto-generated by `lmx build` from config.yaml. Follows standard Unix `--help` conventions.

```
summarize - Summarize text or documents concisely

Usage: summarize [OPTIONS] [INPUT]

Input can be provided as an argument or piped via stdin.

Options:
  --max-chars <number>    Maximum characters in the summary output (default: 500)
  --format <string>       Output format: paragraph, bullets, oneliner (default: paragraph)
  --instructions <string> Additional instructions for the summarization
  --model <provider/id>   Override the LLM model
  --temperature <float>   Override temperature
  --verbose               Show runner diagnostics on stderr
  --help                  Show this help text

Tested models:
  anthropic/claude-sonnet-4-6   score: 0.95 (recommended)
  openai/gpt-5.4                score: 0.88
  google/gemini-2.5-pro         score: 0.82
```

The "Tested models" section is populated from bench results when available.

## Parameter Handling

### Three Levels

**Runner-level params** (universal, consumed by `lmx`, never reach the LLM):
- `--model <provider/model>` — override model
- `--temperature <float>` — override temperature
- `--verbose` — diagnostics to stderr
- `--help` — print help.txt

**Program-level params** (defined in config.yaml, injected into LLM context):
- Defined per-program in config.yaml
- Parsed and validated by the runner based on config.yaml definitions
- Effective values are assembled from defaults plus user overrides
- Injected into context so the LLM sees the actual values in effect

**Input** (positional args or stdin):
- `summarize "some text"` — positional
- `echo "text" | summarize` — stdin
- Both arrive as the Input section of the context

### Context Assembly

The runner assembles the full context sent to the LLM:

**When params are passed** (`summarize --max-chars 200 --format bullets "some text"`):

```
[LMX.md content]

Parameters:
- max-chars (Maximum characters in the summary output): 200 (user)
- format (Output format: paragraph, bullets, oneliner): bullets (user)

Input:
some text
```

**When no params are passed** (`summarize "some text"`):

```
[LMX.md content]

Parameters:
- max-chars (Maximum characters in the summary output): 500 (default)
- format (Output format: paragraph, bullets, oneliner): paragraph (default)

Input:
some text
```

The runner injects effective param values. User-supplied values override defaults. Params with no default are included only when explicitly provided. This keeps `config.yaml` as the single source of truth while letting the LLM see the actual runtime configuration.

### Validation Boundary

The runner is responsible for CLI/runtime correctness:

- Parsing flags and input sources
- Validating param types, enums, and required values from `config.yaml`
- Validating provider/model selection
- Wiring Pi tools and companion scripts
- Reporting diagnostics to stderr and setting the correct exit code

The runner is NOT responsible for enforcing program-specific output semantics. If an LMX program defines a param like `max-chars`, that is part of the program contract owned by the LMX author and implemented by the prompt and any companion scripts or hooks the program uses.

## Failure Semantics

LMX keeps exit behavior simple:

- `0` — success
- `1` — runtime failure while executing the program
- `2` — usage, config, or validation error before execution

Examples:

- Invalid flag, invalid enum value, invalid program config, or missing required input -> `2`
- Auth failure, model/provider failure, or companion tool/script failure -> `1`
- Successful run with normal output -> `0`

`stdout` is reserved for program output. `stderr` is reserved for diagnostics.

## Benchmarking

### bench.yaml

```yaml
tests:
  - name: "basic-summary"
    input: "The cat sat on the mat. The cat was happy. The mat was red. The cat liked the red mat very much."
    params:
      max-chars: 50
    assert:
      - contains: "cat"
      - max_chars: 50

  - name: "bullets-format"
    input: "Point one is about X. Point two is about Y. Point three is about Z."
    params:
      format: bullets
    assert:
      - contains: "-"
      - min_lines: 2

  - name: "stdin-pipe"
    stdin: true
    input_file: "bench/fixtures/article.txt"
    assert:
      - max_chars: 500
      - not_contains: "Here is a summary"
```

### Assertion Types

**Deterministic** (no LLM needed):
- `contains` / `not_contains` — substring check
- `max_chars` / `min_chars` — length bounds
- `max_lines` / `min_lines` — line count bounds
- `matches_regex` — regex match
- `exit_code` — expected exit code
- `stderr_empty` — no stderr output

**LLM-judged** (uses a judge model, flagged as higher-variance):
- `sentiment` — positive/negative/neutral
- `topic_relevant` — output relates to input topic
- `factual_to_input` — no hallucinated facts beyond input
- `language` — output is in expected language

### Running Benchmarks

```bash
# Bench all configured models
lmx bench summarize

# Bench specific model
lmx bench summarize --model anthropic/claude-sonnet-4-6

# Bench all programs
lmx bench --all

# Results written to bench/results/<provider>_<model>.yaml
```

### Result Files

```yaml
# bench/results/anthropic_claude-sonnet-4-6.yaml
model: anthropic/claude-sonnet-4-6
timestamp: 2026-04-10T15:30:00Z

tests:
  basic-summary:
    pass_rate: 1.0
    avg_latency_ms: 1200
    scores: [1.0, 1.0, 1.0]
  bullets-format:
    pass_rate: 0.93
    avg_latency_ms: 1400
    scores: [1.0, 0.8, 1.0]
overall:
  pass_rate: 0.95
  score: 0.95
```

## CLI Commands

```bash
# Scaffold a new LMX program
lmx create <name>

# Build: generate help.txt, shim, validate config
lmx build <name>

# Install launchers into the configured bin dir
lmx install <name|path>
lmx install --all

# Benchmark
lmx bench <name>
lmx bench <name> --model <provider/model>
lmx bench --all

# Run directly (normally users just call the command by name)
lmx run <program-dir> [args...]

# List available programs
lmx list
```

## Future Integrations

OpenClaw integration is intentionally out of scope for LMX v1.

Future integration may expose installed LMX programs to other agents, keep tool listings in sync, or manage PATH/setup automatically. None of that changes the core LMX design: the runtime contract stays Pi-only and the user-facing interface stays standard Unix commands.

## Directory Layout

```
~/.lmx/
├── config.yaml          # Global lmx config (default model, paths, etc.)
├── programs/            # Local developer-created LMX program folders
│   ├── translate/
│   └── extract-emails/
└── ...

Bundled built-ins live inside the installed `lmx` package under `built-ins/`.
Standalone LMX programs can also be distributed as their own npm packages.
`lmx install` writes per-program launchers into the configured `bin_dir`, which the user can add to PATH.
```

## Design Principles

1. **A program is a folder.** Everything the program needs is self-contained.
2. **config.yaml is the single source of truth.** Param definitions flow to help.txt, runtime context, and benchmarks from one place.
3. **The runner is invisible.** Users invoke commands by name. `lmx` is plumbing.
4. **The model is like a compiler.** Programs declare a default, can be overridden, and benchmarks report compatibility.
5. **Benchmarks over tests.** LLMs are non-deterministic. We measure variance, not assert equality.
6. **Unix contract.** stdin/stdout/stderr, exit codes, composability. No special protocols.
7. **Lean on existing auth.** Reuse Pi credential stores instead of inventing new auth.
